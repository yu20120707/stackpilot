# session-cache Redis timeout 排查

适用场景：

- `session-cache`
- `redis timeout`
- 登录态异常、验证码失败、购物车读写慢
- 线程里出现 `Redis`、`timeout`、`连接池`、`session-cache`

优先排查顺序：

1. 检查 Redis 实例 CPU、内存、慢查询、连接数和命中率。
2. 核对 `session-cache` 应用实例的连接池配置是否被改动，是否出现池耗尽或重试风暴。
3. 确认是否只有单机房、单分片或单租户超时。
4. 抽样查看超时日志，区分是建连超时、命令执行超时还是网络读写超时。
5. 对照最近变更，检查是否引入了大 key、批量操作、热 key 或序列化膨胀。

判断建议：

- 如果 Redis 端监控同时出现 RT 上升和连接数打满，应优先怀疑依赖侧瓶颈，而不是直接归因业务代码。
- 如果只有 `session-cache` 新版本实例报 timeout，当前判断应偏向“发布或配置变更相关”。
- 如果线程里没有 Redis 指标和超时日志，应把“缺少依赖监控和错误样本”列为缺失信息。

建议补充的证据：

- Redis 慢查询与连接数
- `session-cache` 超时日志
- 连接池配置
- 最近配置变更
- 受影响功能范围
