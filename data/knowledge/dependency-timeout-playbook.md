# 依赖超时与下游异常排查

当线程里出现“timeout”“超时”“下游异常”“数据库”“Redis”“MQ”“连接池”时，可优先检查：

- 数据库连接池是否打满，慢 SQL 是否增加
- Redis、MQ、HTTP 下游接口是否出现 timeout 或错误码抬升
- 线程池、协程池、连接池是否耗尽
- 单个机房、单个节点、单个租户是否集中报错
- 请求链路中的 trace_id 是否能串出共同失败节点

常见判断模式：

- 如果上游服务报 timeout，但自身 CPU 与内存稳定，优先怀疑下游依赖或网络路径。
- 如果数据库 RT 与连接数同时上升，且业务错误集中在读写接口，优先补充 SQL、连接池和慢查询证据。
- 如果只有部分接口失败，要补充接口名、错误码、样本日志，而不是笼统写“服务异常”。
