# order-service 数据库连接池耗尽排查

适用场景：

- `order-service`
- `db timeout`
- `连接池`
- 下单接口 RT 抬升、超时或 5xx

优先排查顺序：

1. 查看 `order-service` 实例侧数据库连接池使用率、等待时间、超时次数。
2. 检查数据库实例 CPU、活跃连接数、慢 SQL、锁等待和事务堆积。
3. 核查最近是否有大事务、批量补偿、索引变更、热点订单查询。
4. 对照接口维度，确认是否集中在下单、支付确认、订单查询或退款路径。
5. 如果线程里提到“发布后 RT 上升”，同时核对 ORM 配置、SQL 改动和连接池参数改动。

判断建议：

- 如果应用侧连接池等待时间和数据库活跃连接数同时升高，可先判断为“数据库链路拥塞”。
- 如果只有个别接口受影响，摘要中应写明“问题可能集中在订单写路径或特定 SQL”。
- 若没有慢 SQL、连接池指标或错误日志，必须把这些列为缺失信息。

建议补充的证据：

- 连接池监控截图或数值
- 慢 SQL 样本
- 下单 / 退款接口错误日志
- 最近数据库或 ORM 相关变更
- 影响时段与恢复时段
