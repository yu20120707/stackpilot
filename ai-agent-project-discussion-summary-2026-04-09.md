## 3. 做了什么关键判断

### 3.1 关于你的个人背景

我们先看了你的简历，判断你更适合把项目做成：

- `AI 增强型后端系统`
- 而不是 `花哨的多 Agent Demo`

原因是你的经历更偏：

- C++ / 后端
- IM / 微服务
- 性能优化
- 缓存与降级
- AI 摘要落地
- 日志分析

这些能力更适合往：

- 研发协作助手
- 故障处理助手
- AI 知识辅助系统
- LLM 工程化 / 可观测性

这种方向去转。

### 3.2 关于 GitHub 参考项目

我们判断 GitHub 上的现成项目不能“整套照搬”，应该按模块拆开借鉴：

- `Onyx`
  适合参考企业知识助手的系统边界、连接器、worker、RAG 分层
- `Langfuse`
  适合参考 tracing、eval、prompt 管理和可观测性
- `LiteLLM`
  适合参考模型接入层、fallback、统一兼容接口
- `RAGFlow`
  适合参考知识入库、chunking、引用溯源、混合检索
- `LangGraph`
  只适合做内部工作流参考，不适合当求职项目的标题和卖点

结论是：

`不要抄一个仓库，要按系统模块借。`

### 3.3 关于 `multi-agent-meeting-assistant`

我们评估后给出的结论是：

- `能参考代码骨架`
- `不适合直接作为求职主项目原样使用`

可借的部分：

- 服务入口层
- LLM adapter
- Jira / 飞书集成
- 结构化提取思路
- 工作流骨架

不适合沿用的部分：

- 会议 / 音频 / 转写场景
- 多 Agent 叙事
- 三语言展示
- 明显偏面试包装的 README / 计划说明

结论是：

`借代码骨架，不借项目故事。`

### 3.4 关于面试策略

我们明确了一条很重要的策略：

`项目要往“系统设计题”方向靠，不要往“框架实现题”方向靠。`

也就是：

- 不要把项目包装成“多 Agent 平台”
- 不要把卖点写成 LangGraph / LangChain / Prompt 技巧
- 要把它写成“一个真实研发系统里的 AI 增强模块”

这样面试官更可能问：

- 为什么这样分层
- 为什么用 Redis / MySQL / 向量检索
- 为什么用混合检索
- 为什么要做结构化输出
- 为什么要做降级、缓存、幂等、状态机

而不是盯着你问：

- 每个 agent 的 prompt 怎么写
- 工作流节点怎么调度
- 具体框架 API 怎么用

## 4. 最终收敛出的项目方向

经过多轮讨论，最终把项目方向收敛为：

`飞书入口的研发故障处理与知识辅助系统`

一句话定位：

`不替代飞书，而是在飞书之上补足研发场景中的跨系统上下文聚合、知识追溯和任务闭环能力。`

这个方向优于“会议助手”或“通用聊天机器人”，原因是：

1. 更贴近真实开发场景
2. 和飞书已有功能错开
3. 更像后端系统，而不是 AI Demo
4. 更容易讲系统设计、存储分层、工程取舍
5. 更符合你现有的 IM、日志、缓存、AI 摘要背景

## 5. 最终确定的项目主链路

项目主链路已经明确固定为：

`飞书事件 -> 上下文聚合 -> 知识检索 -> 结构化结论 -> Jira / 飞书闭环`

这条链路是后续实现、写简历、面试表达的统一主线。

具体可展开为：

1. 飞书群内出现报警消息或人工触发事件
2. 系统创建 `event_id`
3. 聚合飞书讨论、Jira、发布记录、历史故障、FAQ、值班手册、日志摘要
4. 基于短期记忆和长期知识进行 RAG 检索与回答
5. 输出结构化结果：
   - 事件摘要
   - 影响范围
   - 可能原因
   - 排查建议
   - 待办项
6. 把待办或风险项同步到 Jira / 飞书
7. 最后可生成复盘草稿

## 6. 已做出的关键决策

### 6.1 做什么

项目必须 cover 的方向已经确定：

1. 飞书入口
2. 上下文聚合
3. Redis 短期记忆
4. 长期知识库
5. Query Rewriting
6. BM25 + 向量混合检索
7. 结构化输出
8. Jira / 飞书闭环
9. 状态机、幂等、重试、降级、日志与基础可观测性

### 6.2 不做什么

项目明确不做这些方向：

1. 多 Agent 编排作为主卖点
2. 会议音频转写
3. 通用聊天机器人
4. 自研模型 / 自研向量库 / 自研 Agent 框架
5. 复杂 planner / autonomous agent
6. 花哨前端
7. 多模态
8. 三语言实现

### 6.3 技术上的取舍

技术路线也已经收敛：

- 飞书负责入口和通知，不做前台主系统
- Redis 负责短期记忆、缓存、去重
- MySQL 负责任务状态、结果记录、知识元数据
- 向量检索负责 embedding 检索
- BM25 负责关键词 / 倒排召回
- LLM 接入走 OpenAI 兼容接口或 LiteLLM
- 观测先做应用日志 + 业务埋点，时间够再接 Langfuse

同时也明确了：

`不要把框架名写成项目卖点。`
