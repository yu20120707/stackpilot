# Live Feishu Robot Prompts

下面这些内容可以直接复制到飞书线程里做手工测试。

使用方式：

1. 先在同一条线程里连续发送“场景上下文”里的 3 到 5 行内容
2. 最后再发送“触发语句”
3. 观察回帖是否出现：
   - 当前判断
   - 已知事实
   - 影响范围
   - 下一步建议
   - 参考来源

## 场景 1：payment-api 发布后 5xx

场景上下文：

```text
payment-api 在 23:10 发布后开始出现 5xx 报警。
23:14 已经执行回滚，错误率从 12% 降到 4%，但还没有完全恢复。
目前还没拿到完整错误日志，受影响接口和租户范围还没确认。
线程里怀疑和这次支付路由改动有关。
```

触发语句：

```text
@stackpilot 分析一下报警原因
```

期望更容易命中的知识来源：

- `payment-api 发布回滚与 5xx 排查`
- `故障分析信息补充清单`

## 场景 2：session-cache Redis timeout

场景上下文：

```text
session-cache 从 14:02 开始出现 Redis timeout，登录态偶发失效。
当前只在华东机房明显，连接池告警也在升高。
最近刚调大过 session-cache 的并发和连接池配置。
还缺 Redis 慢查询和超时错误日志样本。
```

触发语句：

```text
@stackpilot 帮我分析一下报警原因
```

期望更容易命中的知识来源：

- `session-cache Redis timeout 排查`
- `故障分析信息补充清单`

## 场景 3：order-service 数据库连接池耗尽

场景上下文：

```text
order-service 在晚高峰开始出现 db timeout，下单 RT 明显升高。
连接池等待时间升高，但现在还没拉到慢 SQL。
退款接口目前正常，问题更像集中在下单写路径。
最近发布改过 ORM 配置和部分订单查询 SQL。
```

触发语句：

```text
@stackpilot 分析一下这次故障
```

期望更容易命中的知识来源：

- `order-service 数据库连接池耗尽排查`
- `故障分析信息补充清单`

## 场景 4：补充信息后重试

先延续场景 1 的线程，再补充：

```text
刚拿到 payment-api 错误日志，主要是 settlement-service 返回 502。
受影响接口集中在支付确认接口，租户范围暂时还不明确。
```

触发语句：

```text
@stackpilot 基于最新信息重试
```

## 场景 5：让机器人做结论总结

在一个已有较长讨论线程底部发送：

```text
@stackpilot 帮我总结当前结论
```
